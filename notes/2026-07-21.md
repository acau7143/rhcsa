## 8. AutoFS (Create and configure file systems)

### Question
```
8.
AutoFS  (Create and configure file systems)
1. Install the necessary NFS and Autofs packages.
2. Configure Autofs to mount the remote directory 192.168.1.100:/shares/public
   (replace IP with your lab's NFS server IP) to the local directory /data/public.
3. The mount should only happen on-demand (when the directory is accessed).
4. Ensure Autofs starts automatically at boot.
```

### My Solution

**infra-server (192.168.56.101) — NFS 서버 준비 (연습용 배경 세팅, 시험 채점 대상 아님)**
```bash
mkdir -p /shares/public
chmod 755 /shares/public

vim /etc/exports
# /shares/public   192.168.56.0/24(rw,sync,no_root_squash)

exportfs -ra
systemctl enable --now nfs-server

firewall-cmd --add-service={nfs,rpc-bind,mountd} --permanent
firewall-cmd --reload
```

**exam-vm (192.168.56.102) — autofs 클라이언트 설정 (실제 시험 채점 대상)**
```bash
dnf install autofs nfs-utils -y

vim /etc/auto.master
# /data   /etc/auto.data

vim /etc/auto.data
# public  -rw,sync  192.168.56.101:/shares/public

systemctl enable --now autofs
```

### Verification
```bash
[root@exam-vm ~]# cd /data/public
[root@exam-vm public]# ls
testfile
[root@exam-vm public]# mount | grep public
192.168.56.101:/shares/public on /data/public type nfs4 (rw,relatime,sync,vers=4.2,rsize=262144,wsize=262144,...)
```
- infra-server에서도 `/shares/public/testfile` 확인 → 크로스 검증 완료 (실전에서는 서버 접근 권한이 없으므로 이 단계는 생략, `mount`/`findmnt` 결과 자체로 충분)

### 내가 헷갈렸던 것 / 배운 것
- `firewall-cmd --add-service=nfs`만 열면 되는 줄 알았는데, `showmount`가 쓰는 **MOUNT 프로토콜(rpcbind 111번, mountd)**은 별도 서비스로 열어야 함. `nfs` 서비스(2049)만으로는 `showmount -e`가 `clnt_create: RPC: Unable to receive`로 실패.
- 방화벽 수정 명령을 클라이언트(exam-vm)와 서버(infra-server) 중 **엉뚱한 쪽에 적용**했던 실수 있었음 → `hostnamectl set-hostname`으로 프롬프트를 미리 구분해두는 습관이 중요함을 체감.
- `/data/public` 경로는 마스터맵의 상위 경로(`/data`) + 데이터맵의 key(`public`)를 autofs가 자동으로 이어붙여 만든 것 — 수동 `mkdir` 불필요.
- 마운트 여부와 "파일이 실제로 존재하는가"는 별개 개념. 언마운트돼도 서버의 실제 파일은 그대로 있고, 클라이언트에서만 "안 보이게" 될 뿐.
- 언마운트(타임아웃) 테스트하려면 반드시 `cd ~`로 그 디렉터리 밖으로 나가야 함. 터미널이 그 안에 머물러 있으면 busy 상태로 인식돼 타임아웃이 안 걸림.
- **실제 RHCSA 시험에서는 NFS 서버 구성 자체가 채점 대상이 아님** (RHCSA는 클라이언트 관점 스킬만 요구, 서버 구성은 RHCE 영역). 오늘 infra-server에서 한 작업은 순수 연습 환경 구축용.

## 2-1. Root Password 재설정 (Node2 · Operate running systems)

### Question
```
Node 2
1. [필수] Root Password 재설정 또는 응급 복구 진입 (Operate running systems)

a. Root Password 재설정
   Reboot → GRUB 메뉴에서 e 누르기
   linux … 맨 끝에 : rw init=/bin/bash 입력 (rw도 같이 적어주기)
   ctrl+x 누르기
   passwd root
   touch /.autorelabel
   exec /sbin/init
```

### My Solution
**infra-server에서 진행 (Node2 역할)**
```
1. reboot
2. GRUB 메뉴에서 커서 이동 후 e
3. linux ... 줄 맨 끝에 추가: rw init=/bin/bash
4. Ctrl+X (부팅 진행)
5. bash 프롬프트에서:
   passwd root
   touch /.autorelabel
   exec /sbin/init
```

### Verification
```bash
[reallinux@infra-server ~]$ sestatus
SELinux status:                 enabled
Current mode:                   enforcing
Mode from config file:          enforcing

[reallinux@infra-server ~]$ getenforce
Enforcing
```
- 새 비밀번호로 로그인 성공, autorelabel 이후 SELinux enforcing 정상 복귀 확인.

### 내가 헷갈렸던 것 / 배운 것
- `init=/bin/bash`만 쓰면 루트 파일시스템이 `ro`로 마운트되어 `passwd`가 `Authentication token manipulation error`로 실패할 수 있음. 커널 파라미터에 `rw`를 같이 적으면(`rw init=/bin/bash`) 별도 `mount -o remount,rw /` 단계 없이 한 번에 해결됨.
- `touch /.autorelabel`을 빠뜨리면 SELinux 라벨이 깨진 채로 남아, 나중에 로그인 시 PAM이 `/etc/shadow` 접근을 거부해 인증 실패할 수 있음. 원인이 "권한 문제(rwx)"가 아니라 "SELinux 라벨 문제"라는 걸 구분하는 게 핵심.
- `Ctrl+X`(부팅 진행) 대신 Enter를 누르면 편집이 반영 안 되고 GRUB 메뉴로 되돌아감.
- `exec /sbin/init`으로 정상 부팅 프로세스를 이어받아야 함 — bash가 PID 1인 상태라 일반 `reboot` 명령은 정상 동작하지 않을 수 있음.
- 이 실습은 GRUB 단계라 **man 페이지 참고가 불가능**해서, 절차 자체를 암기해둬야 하는 몇 안 되는 예외 항목.

## 보충. GRUB 부팅 메뉴 타임아웃 변경

### Question
(덤프 원문 없음 — 보충 학습 주제로 직접 진행)

### My Solution
```bash
vim /etc/default/grub
# GRUB_TIMEOUT=10

grub2-mkconfig -o /boot/grub2/grub.cfg
```

### Verification
```bash
[root@infra-server reallinux]# grep GRUB_TIMEOUT /etc/default/grub
GRUB_TIMEOUT=10

[root@infra-server reallinux]# grep timeout /boot/grub2/grub.cfg
  set timeout=10
  set timeout=10
  ... (fallback/once-menu 관련 조건문 다수)
```

### 내가 헷갈렸던 것 / 배운 것
- `/etc/default/grub`은 사람이 편집하는 **설정 소스**일 뿐이고, 실제 부팅 시 GRUB이 읽는 파일은 `/boot/grub2/grub.cfg`. `grub2-mkconfig`를 안 돌리면 값을 아무리 고쳐도 반영 안 됨 — 가장 흔한 함정.
- `grub.cfg` 안에 `timeout` 값이 여러 번(10, 60, 0, 1 등) 나와서 당황했는데, **파일 상단의 값이 실제 적용값**이고 나머지는 GRUB 내부 fallback/once-menu 로직용 조건문. 값이 여러 개 보인다고 잘못 설정된 게 아님.
- BIOS/UEFI에 따라 `grub.cfg` 경로가 다름 (`/boot/grub2/grub.cfg` vs `/boot/efi/EFI/rocky/grub.cfg`) — `ls /sys/firmware/efi` 존재 여부로 사전 확인 필요.

## 원래 상태로 되돌리는 명령어 (Rollback)

**autofs (exam-vm)**
```bash
systemctl disable --now autofs
vim /etc/auto.master   # /data 줄 삭제
rm -f /etc/auto.data
```

**NFS 서버 (infra-server)**
```bash
systemctl disable --now nfs-server
vim /etc/exports   # /shares/public 줄 삭제
exportfs -ra
firewall-cmd --remove-service={nfs,rpc-bind,mountd} --permanent
firewall-cmd --reload
```

**root 비밀번호**: 되돌릴 필요 없음 (의도된 변경, 그대로 유지)

**GRUB 타임아웃**
```bash
vim /etc/default/grub   # GRUB_TIMEOUT=5 (기본값)로 복원
grub2-mkconfig -o /boot/grub2/grub.cfg
```
