## 1-4. User Account Creation

### Question
```
Create User accounts with supplementary group

- create the group named "sysadms"
- create users named "natasha" and "harry", will be the supplementry group "sysadms"
- create a user named "sarah", should have non-interactive shell and it should
  be not the member of "sysadms"
- password for all users should be "trootent"
- "sysadms" group has access to user add in the server
- "harry" user has access to set password for users without asking sudo password
  (이건 출제 안 됨, 혹시 몰라 준비)
```

### My Solution
```bash
groupadd sysadms

useradd -G sysadms natasha
useradd -G sysadms harry
useradd -s /sbin/nologin sarah

passwd natasha
passwd harry
passwd sarah
# 비밀번호: trootent (직접 타이핑 방식 사용, --stdin은 결과가 조용히 실패할 수 있어 대화형 선택)

visudo
# 아래 두 줄 추가:
# %sysadms ALL=(ALL) NOPASSWD: ALL
# harry ALL=(ALL) NOPASSWD: /usr/bin/passwd
```

### Verification
```bash
tail -5 /etc/passwd
getent group sysadms
su - harry
sudo -l
```
결과:
```
natasha:x:1001:1002::/home/natasha:/bin/bash
harry:x:1002:1003::/home/harry:/bin/bash
sarah:x:1003:1004::/home/sarah:/sbin/nologin

User harry may run the following commands on localhost:
    (ALL) NOPASSWD: ALL
    (ALL) NOPASSWD: /usr/bin/passwd
```

### 내가 헷갈렸던 것 / 배운 것
- **`-G`(대문자) vs `-g`(소문자)**: `-G`는 보조 그룹 추가, `-g`는 기본 그룹 자체를 교체. 문제에 "supplementary group"이라고 명시되어 있으면 무조건 대문자 `-G`.
- **`echo | passwd --stdin` vs `passwd`**: `--stdin`은 자동화엔 빠르지만 결과가 눈에 잘 안 보여서 조용히 실패할 위험이 있음. 시험처럼 확실한 검증이 필요한 상황에선 대화형 `passwd`가 더 안전.
- **`harry`로 로그인해도 root가 되는 게 아님**: `harry`가 `sudo` 없이 명령을 치면 여전히 일반 사용자 권한. `sudo`를 붙였을 때만 그 순간 root 권한을 빌리는 방식.
- **`sudo -l` 결과에 두 줄이 뜬 이유**: `harry`는 `sysadms` 그룹 소속(그룹 규칙 적용) + 개인 사용자로도 등록(개인 규칙 적용), 두 규칙이 합산되어 보임. 사실 `harry`는 그룹 규칙(ALL) 하나만으로 이미 `passwd`도 포함되므로, 개인 규칙은 실질적으로 중복.
- **미완료**: 실제로 `sudo` 명령 실행 시 비밀번호 프롬프트가 안 뜨는지까지는 확인 못 함 (테스트 계정 삭제로 흐름 중단됨) → 재실습 필요.

## 1-13. User Creation with Specific UID

### Question
```
Create a user "unilao" with UID "2334" with password as "souspolo"
```

### My Solution
```bash
useradd -u 2334 unilao
passwd unilao
# 비밀번호: souspolo
```

### Verification
```bash
tail -1 /etc/passwd
id unilao
```
결과:
```
unilao:x:2334:2334::/home/unilao:/bin/bash
uid=2334(unilao) gid=2334(unilao) groups=2334(unilao)
```

### 내가 헷갈렸던 것 / 배운 것
- **UID를 왜 강제 지정하나**: 여러 서버를 NFS로 묶어 쓸 때, 같은 사용자가 서버마다 UID가 다르면 파일 소유권이 꼬임. 그래서 UID를 고정해서 서버 간 동일하게 맞추는 게 실무 이유.
- **`id` 명령어 결과 해석**: `uid=`, `gid=`, `groups=` 세 개가 나오는데, `useradd -u`만 쓰면 개인 그룹(GID)도 UID와 같은 번호로 자동 생성됨 (2334/2334 동일하게 뜬 이유).
- **UID 중복 시**: `useradd: UID 2334 is not unique` 에러 발생, `-o` 옵션으로 강제 허용 가능 (참고로만 익힘).

## 1-6. Collaborative Directory (SGID)

### Question
```
Create a collaborative Directory

- Create the Directory "/home/manager" with the following characteristics
- Group ownership of "/home/manager" should go to "sysadms" group
- The directory should have full permission for all members of "sysadms" group
  but not to the other users except "root"
- Files created in future under "/home/manager" should get the same group ownership
```

### My Solution
```bash
mkdir /home/manager
chown :sysadms /home/manager
chmod 2770 /home/manager
```

### Verification
```bash
ls -ld /home/manager
su - natasha
mkdir /home/manager/t
touch /home/manager/test
ls -l /home/manager/
```
결과:
```
drwxrws---. 2 root sysadms 6 Jul  2 00:46 /home/manager/

drwx--S---. 2 natasha sysadms 6 Jul  2 01:06 t
-rw-------. 1 natasha sysadms 0 Jul  2 01:05 test
```

### 내가 헷갈렸던 것 / 배운 것
- **`chown :sysadms`의 문법**: 콜론 앞이 비어있으면 소유자는 그대로 두고 그룹만 바꿈. `chgrp sysadms /home/manager`와 완전히 동일한 효과.
- **`chown -R`**: 지금 문제는 디렉토리 자체만 바꾸면 되지만, "디렉토리와 그 안의 모든 파일"이라고 나오면 `-R`(재귀) 필수라는 걸 알게 됨.
- **SGID(`2770`) 동작 원리**: 앞자리 `2`가 SGID 비트. 이게 걸린 디렉토리 안에서 만들어지는 파일/하위디렉토리는 만든 사람과 상관없이 무조건 그 디렉토리의 그룹(sysadms)을 상속받음. natasha가 만든 `t`, `test`가 natasha 개인 그룹이 아니라 sysadms로 뜬 게 그 증거.
- **`d`와 `S`는 서로 무관한 자리**: `ls -l` 출력의 1번째 글자(`d`)는 파일 종류(디렉토리) 표시일 뿐이고, SGID 표시(`s`/`S`)는 5~7번째(그룹 권한) 자리 중 실행권한 위치에 별도로 나타남. 완전히 다른 자리라 헷갈리면 안 됨.
- **대문자 `S` vs 소문자 `s`**: `s`(소문자)=SGID + 그룹 실행권한(x) 둘 다 있음. `S`(대문자)=SGID는 있는데 그룹 실행권한(x)이 없음. 지금 `t` 디렉토리가 `--S`로 뜬 건, umask 077 때문에 그룹 실행권한 자체가 원래 없어서 생긴 정상적인 표시 (설정 실수 아님).

## 1-16. umask Setting for Specific User

### Question
```
사용자 natasha가 파일과 디렉터리를 생성했을 때, 다음과 같은 권한이 설정되도록 구성하시오

- 파일 생성 시 기본 권한: rw-------
- 디렉터리 생성 시 기본 권한: rwx------
```

### My Solution
```bash
vi /home/natasha/.bash_profile
# 맨 아래 추가:
# umask 077
```

### Verification
```bash
su - natasha
umask
touch testfile
mkdir testdir
ls -ld testfile testdir
```
결과 (앞서 SGID 테스트에서 확인한 것과 동일 사용자로 검증됨):
```
-rw-------. 1 natasha sysadms 0 ... test
drwx--S---. 2 natasha sysadms 6 ... t   (소유자 권한 rwx까지 확인, 그룹 이하는 SGID 영향)
```

### 내가 헷갈렸던 것 / 배운 것
- **umask 계산 방식**: 실제 권한 = 최대 기본값(파일 666, 디렉토리 777) − umask. `rw-------`(600)와 `rwx------`(700)를 동시에 만족하는 값이 `077`.
- **`umask 077`을 터미널에 그냥 치면 전체 사용자한테 적용되는 줄 알았음** → 아니었음. 터미널에서 직접 치면 **그 세션에만** 일시 적용되고 로그아웃하면 사라짐. 특정 사용자에게 영구 적용하려면 `.bash_profile`처럼 그 사용자 로그인 시 실행되는 파일에 써야 함. `/etc/profile`에 넣어야 전체 사용자 적용.

## 1-14. chage — Password Max Days

### Question
```
Set Password Max days as 20 days
```

### My Solution
```bash
chage -M 20 natasha
```

### Verification
```bash
chage -l natasha
```

### 상태
**미완료** — 명령 실행 후 결과 확인을 아직 안 함. 다음 세션에서 `chage -l natasha` 출력의 `Maximum number of days between password change: 20` 확인 필요.

### 참고 개념
- `chage -M`(대문자) = 최대 사용 기간, `chage -m`(소문자) = 최소 사용 기간. 대소문자 반대로 쓰면 정반대 정책이 걸리므로 주의.
- 전역으로 신규 계정 전체에 적용하려면 `/etc/login.defs`의 `PASS_MAX_DAYS`를 수정 (단, 기존 계정에는 영향 없음, 앞으로 생성되는 계정에만 적용).

## 부가: userdel 관련 실수

### 겪은 상황
```bash
userdel natasha        # -r 없이 삭제
useradd -G sysadms natasha   # 재생성 시도
```
```
useradd: warning: the home directory /home/natasha already exists.
useradd: Not copying any file from skel directory into it.
Creating mailbox file: File exists
```

### 원인 / 배운 것
`userdel`만 쓰면 계정 정보(`/etc/passwd` 등)만 삭제되고 홈 디렉토리는 남는다. 그래서 같은 이름으로 재생성할 때 "이미 홈 디렉토리가 있다"는 경고가 뜬 것. 계정+홈 디렉토리까지 완전히 지우려면 `-r` 옵션 필수.

```bash
userdel -r natasha
```

## 원래 상태로 되돌리는 명령어 (Rollback)

```bash
# 1. 사용자 삭제 (-r로 홈 디렉토리까지 정리)
userdel -r natasha
userdel -r harry
userdel -r sarah
userdel -r unilao

# 2. 그룹 삭제 (사용자 삭제 후 진행)
groupdel sysadms

# 3. 협업 디렉토리 삭제
rm -rf /home/manager

# 4. 정리 확인
tail -10 /etc/passwd
getent group sysadms   # 에러 나오면 정상 삭제
ls /home/
```

sudoers에 추가한 규칙은 사용자/그룹을 지워도 `/etc/sudoers`에 텍스트로 남아있으므로, 필요하면 직접 정리:
```bash
visudo
# 아래 두 줄 삭제
# %sysadms ALL=(ALL) NOPASSWD: ALL
# harry ALL=(ALL) NOPASSWD: /usr/bin/passwd
```
