## 0. Day 6 복습 중 나온 추가 질문 — ro/rw

Day 6 `init=/bin/bash` 복구 절차 복습하다가 "ro가 뭔데?"라는 질문이 나와서 별도로 정리.

- `ro`(read-only)는 **파일시스템 마운트 옵션**. 그 마운트 전체에 쓰기 시스템콜을 커널 단에서 막는 것.
- `init=/bin/bash`로 부팅하면 정상 부팅 절차(fsck 검사 후 rw 마운트)를 건너뛰기 때문에, 커널이 안전을 위해 기본값으로 ro만 올려놓음.
- 그래서 `passwd`가 `/etc/shadow`에 쓰려면 `mount -o remount,rw /`로 명시적으로 풀어줘야 함.
- 헷갈린 점: `chmod`(파일 단위 권한)와 `ro`(파일시스템 전체 단위 마운트 옵션)는 레벨이 다름.
- 증상: ro 상태에서 쓰기 시도하면 `Read-only file system` 에러 — 이 메시지 보면 제일 먼저 `mount | grep 마운트포인트`로 ro 여부 확인하는 습관.

## 1. Crontab (Deploy, configure, and maintain systems)

### Question
Configure a cron job that runs every 2 minutes and executes: logger "EX200 in progress" as the user natasha (jacques로 대체).
Configure a cron job that runs daily at 12:30 local time and executes /bin/echo "Hello World" as the user natasha (jacques로 대체), 결과를 파일에 저장.
Configure a cron job for root, running daily at 12:45 AM, that finds and deletes all empty files in /tmp.
Configure the system so that only the user tom is allowed to create cron jobs.

### My Solution
```bash
# ① 2분마다 logger
crontab -u jacques -e
*/2 * * * * /usr/bin/logger "EX200 in progress"

# ② daily 12:30 echo (파일 저장) — 최종 수정 안 함, > 로 남아있음 (>> 로 고쳐야 함, 시간도 테스트용으로 임시 변경한 채 미복구)
30 12 * * * /bin/echo "Hello World" > /home/jacques/hello.log

# ③ root, 00:45 /tmp 빈 파일 삭제
crontab -u root -e
45 00 * * * find /tmp -type f -empty -delete

# ④ cron.allow로 tom만 허용
echo tom > /etc/cron.allow
```

### Verification
```
# ①: /var/log/messages 에 실행 기록 확인
Jul 22 02:06:02 localhost jacques[2611]: EX200 in  progress

# ②: hello.log 생성 및 내용 확인 (임시 시간으로 테스트)
Hello World

# ④: jacques 직접 로그인 후 crontab -e 시도
[jacques@localhost ~]$ crontab -e
You (jacques) are not allowed to use this program (crontab)
See crontab(1) for more information
```

### 내가 헷갈렸던 것 / 배운 것 (질문한 것 포함)
- **"각 요소가 뭘 뜻하는지"** → crontab 필드는 `분 시 일 월 요일 명령` 순서 고정. `*`는 "조건 없음", `*/N`은 "N 간격마다". 요일 필드는 0과 7 둘 다 일요일 (1=월요일 아님, 우리나라 감각과 다름).
- 12AM = hour 0, 12PM = hour 12 (처음에 `45 12`로 잘못 씀 → `45 00`으로 정정)
- root가 `crontab -u <user> -e`로 대리 편집하면 cron.allow 체크를 안 받음 (검사는 "실행자" 기준이지 `-u` 대상이 아님)
- `logger` 기본 태그는 실행한 사용자명 → 로그에 `logger`가 아니라 `jacques`로 찍힘
- `cronie`는 패키지명, 서비스명은 `crond` (패키지명 ≠ 서비스명, `systemctl status cronie`는 실패함)

## 2-3. Flatpak (Manage software)

### Question
Configure Flatpak so that user student can install codium from the Flathub repository.
- A flatpak repository has been made available from http://ABC (실제 시험엔 구체적 URL 제공됨)
- The repository should be named flatdb
- The repository must be configured only for the user student
- Install the codium application for user student

### My Solution
```bash
useradd student
passwd student

su - student
flatpak remote-add --if-not-exists --user flatdb https://dl.flathub.org/repo/flathub.flatpakrepo
flatpak search --user codium   # App ID: com.vscodium.codium
flatpak install --user flatdb com.vscodium.codium   # 연습 목적상 다운로드 중단
```

### Verification
```
[student@localhost ~]$ flatpak remotes --user
Name   Options
flatdb
```

### 내가 헷갈렸던 것 / 배운 것 (질문한 것 포함)
- **"나 이게 뭐하는건지 아직도 이해가 안돼"** → dnf와 개념적으로 똑같은 "패키지 설치 도구"인데, dnf가 "동네 마트"라면 Flatpak은 "별도 온라인몰(Flathub)". 마트처럼 그냥 있는 걸 사는 게 아니라, **몰 자체를 먼저 등록(remote-add)**해야 함.
- **"문제랑 url이 다른 이유는?"** → 문제의 `http://ABC`는 실제 시험장에서 주어질 URL의 자리표시자(placeholder)일 뿐. 실제 시험에선 그 자리에 진짜 URL이 주어지므로 외울 필요 없고, `remote-add --user`, `install --user` 같은 **명령어 구조와 옵션**만 기억하면 됨.
- **"remote-add 명령어 요소 하나하나 설명"** → `remote-add`(등록 서브커맨드) + `--if-not-exists`(이미 있으면 에러 없이 넘어감) + `--user`(개인 범위 한정) + `flatdb`(우리가 붙인 이름표) + URL(실제 저장소 위치). remote-add 단계에서는 아직 아무 앱도 안 깔림, "가게 위치만 등록"한 것.
- root로 `--user` 명령을 실행하면 root의 개인 범위(`/root/.local/share/flatpak`)에 등록됨 → 반드시 `su - student` 먼저 해야 함
- `flatpak search`에 `--user`를 안 붙이면 시스템 범위(초기화 안 된 `/var/lib/flatpak`)까지 뒤지다가 `opendir` 에러 발생

## 4. Systemd Timer (Deploy, configure and maintain systems & Create simple shell scripts)

### Question A
On ServerB, replace a legacy Cron job with a Systemd Timer.
1. Create a script /usr/local/bin/notify.sh that prints "Time for Break!" to the system log (use logger). Make it executable.
2. Create a service unit notify.service.
3. Create a timer unit notify.timer that runs this service daily at 7:00 AM.
4. Ensure the timer is active and enabled.

### My Solution
```bash
# /usr/local/bin/notify.sh
#!/bin/bash
logger "Time for Break!"

chmod +x /usr/local/bin/notify.sh
```

```ini
# /etc/systemd/system/notify.service
[Unit]
Description=Notify Break Time Service

[Service]
Type=oneshot
ExecStart=/usr/local/bin/notify.sh
```

```ini
# /etc/systemd/system/notify.timer
[Unit]
Description=Notify Break Time Timer

[Timer]
OnCalendar=*-*-* 07:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

```bash
systemctl daemon-reload
systemctl enable --now notify.timer
```

### Verification
```
[root@localhost ~]# systemctl status notify.timer
● notify.timer - Notify Break Time Timer
     Loaded: loaded (/etc/systemd/system/notify.timer; enabled; preset: disabled)
     Active: active (waiting) since Wed 2026-07-22 04:00:41 EDT
    Trigger: Wed 2026-07-22 07:00:00 EDT; 2h 58min left
   Triggers: ● notify.service

[root@localhost ~]# systemctl list-timers | grep notify
Wed 2026-07-22 07:00:00 EDT 2h 58min left - - notify.timer notify.service
```

### 내가 헷갈렸던 것 / 배운 것 (질문한 것 포함)
- **"이 명령어들은 외워서 가야되는거겠지"** → 정확한 문법(OnCalendar 표기법 등)은 안 외워도 됨. 시험 중 유일하게 참고 가능한 게 man 페이지라서, `man systemd.timer`/`man systemd.time`에 EXAMPLES까지 다 있음. 대신 외워야(몸에 배야) 하는 건 **구조**(.service+.timer 이름 짝 맞추기), **순서**(daemon-reload → enable --now), 그리고 **저장 후 반드시 cat으로 재확인하는 습관**.
- **"WantedBy=timers.target 이건 어디서 보는겨 man"** → `[Timer]` 섹션 전용 옵션은 `man systemd.timer`에 있지만, `[Unit]`/`[Install]` 섹션(Description, WantedBy 등)은 **모든 유닛 타입 공통**이라 `man systemd.unit`에서 찾아야 함. 타이머 전용 man 페이지만 보고 없다고 당황하기 쉬움.
- `OnCalendar` 오타(`OnCalender`)는 systemd가 에러 없이 조용히 무시함 → 저장 후 반드시 `cat`으로 재확인하는 습관 필요
- `daemon-reload` 안 하면 새로 만든 유닛 파일을 systemd가 인식 못함 (`daemon-restart`라는 명령은 존재하지 않음, 오타 주의 — 실제로 `deamon-reload`로도 오타 냈었음, d-a-e-m-o-n 순서 주의)
- `enable --now`는 `.service`가 아니라 반드시 `.timer`에 걸어야 함

## 5. 로그인 메시지 셸 스크립트 (조건문 연습)

### Question
Build an application rhcsa that prints a message when logged in as ablerate user (jacques로 대체).

### My Solution
```bash
# /usr/local/bin/rhcsa
#!/bin/bash
echo "Welcome to RHCSA!"

chmod +x /usr/local/bin/rhcsa
```

```bash
# /home/jacques/.bash_profile 맨 아래 추가
if [ -f /usr/local/bin/rhcsa ]; then
    /usr/local/bin/rhcsa
fi
```

### Verification
```
[root@localhost ~]# su - jacques
Last login: Wed Jul 22 04:45:47 EDT 2026 on pts/1
Welcome to RHCSA!
```

### 내가 헷갈렸던 것 / 배운 것 (질문한 것 포함)
- **".bashrc vs .bash_profile"** → `.bashrc`는 매 셸 진입마다, `.bash_profile`은 로그인 시 1회만 실행. 비유하면 `.bash_profile`은 "현관에서 한 번"(신발 벗기), `.bashrc`는 "방 옮길 때마다". "로그인했을 때"라는 요구사항엔 `.bash_profile`이 정확한 선택.
- `su user`(대시 없음)는 원칙적으로 비로그인 셸 → 로그인 여부는 `su - user`로 명시하고, `echo $0`(`-bash`면 로그인 셸) 또는 `shopt login_shell`로 검증해야 안전 (실제로 `su jacques`도 됐지만 우연일 수 있어서 `su -`로 재검증함)
