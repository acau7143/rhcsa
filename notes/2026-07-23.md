### 1. YUM Repository 설정 (Manage software)

#### Question
```
Configure YUM repos with the given link
BaseOS http://content.example.com/rhel8.0/x86_64/dvd
AppStream http://content.example.com/rhel8.0/x86_64/dvd/AppStream
```

#### My Solution
```bash
vi /etc/yum.repos.d/abc.repo
```
```ini
[BaseOS]
name=BaseOS
baseurl=http://content.example.com/rhel9.0/x86_64/dvd/BaseOS
enabled=1
gpgcheck=0

[AppStream]
name=AppStream
baseurl=http://content.example.com/rhel9.0/x86_64/dvd/AppStream
enabled=1
gpgcheck=0
```
```bash
# 추가: URL로 zsh 패키지 설치
dnf install http://example.com/pub/zsh.rpm -y
```

#### Verification
```bash
dnf clean all
dnf repolist all
```

#### 내가 헷갈렸던 것 / 배운 것
- `/etc/yum.repos.d/*.repo`는 시스템 로컬 파일이라 네트워크로 자동 공유되지 않음 → node1에서 설정해도 node2는 모름. 지문에 "on server1 and server2"처럼 명시되면 각 노드마다 개별로 반복 작업해야 함
- `.repo` 파일은 `[이름] / name / baseurl / enabled / gpgcheck` 5줄 고정 패턴 → 암기해도 되는 수준
- `gpgcheck=0`은 시험용 자체 dvd repo라 서명 검증을 끄는 예외적 상황 (실무에서는 보통 1)

### 2. Network 설정 — Node1 (Manage Basic Networking)

#### Question
```
Host Name: node1.domain250.example.com
IPv4 address: 172.25.250.100
Subnet mask: 255.255.255.0
Gateway: 172.25.250.254
DNS: 172.25.250.254

nmcli device로 현재 연결된 Connection 확인
nmtui → Edit a connection → manual로 변경 → Address, Gateway, DNS 입력
Search Domain은 건드리지 말 것!
Activate a connection
Set system hostname
```

#### My Solution
```bash
# 1. 인터페이스 확인
nmcli device
ip a
```
```
enp0s3 → 10.0.2.15/24   (VirtualBox NAT, SSH 접속과 무관)
enp0s8 → 192.168.56.102/24 (Host-only, 지금 SSH 접속 중인 인터페이스)
```
```bash
# 2. enp0s3에 설정 (SSH 세션 안 끊기는 쪽)
nmtui
#  Edit a connection → enp0s3 → Manual
#  Addresses: 172.25.250.100/24
#  Gateway: 172.25.250.254
#  DNS: 172.25.250.254
#  Search domains: 비워둠
#  Activate a connection → enp0s3 재기동

# 3. hostname 설정
hostnamectl set-hostname node1.domain250.example.com
```

#### Verification
```bash
ip a show enp0s3
nmcli connection show enp0s3 | grep -E "ipv4.addr|ipv4.gateway|ipv4.dns"
hostnamectl status
```

#### 내가 헷갈렸던 것 / 배운 것
- 이더넷이 2개(enp0s3, enp0s8)라 어느 쪽을 건드려야 할지 판단이 필요했음
- 기준: **지금 SSH로 접속해 들어와 있는 인터페이스(enp0s8, Host-only)는 절대 건드리지 않음** → 잘못 만지면 세션 끊김
- 나머지 인터페이스(enp0s3, NAT)가 지문 IP를 설정할 대상
- `Wired connection 1` / `enp0s3` 처럼 연결 이름이 다르게 보이는 건 기능 차이가 아니라 프로파일 이름 차이일 뿐
- Activate 안 하면 설정 저장은 되지만 `ip a`에 반영 안 됨

### 3. NTP 설정 (Deploy, configure and maintain systems)

#### Question
```
Configure NTP - Synchronize the time of your system with the server "ntp.example.com"
```

#### My Solution
```bash
dnf install -y chrony
vi /etc/chrony.conf
```
```
#pool 2.rocky.pool.ntp.org iburst   ← 기존 pool 줄 주석 처리
server ntp.example.com iburst       ← 새로 추가
```
```bash
systemctl enable --now chronyd
```

#### Verification
```bash
systemctl restart chronyd
timedatectl
chronyc sources -v
```

#### 내가 헷갈렸던 것 / 배운 것
- 기본 `chrony.conf`엔 `pool 2.rocky.pool.ntp.org iburst` 한 줄만 활성화되어 있었음 → 이걸 주석 처리하고 `server` 줄 추가하는 게 전부
- `iburst`: 초기 동기화 속도를 높이기 위해 패킷을 몰아서 보내는 옵션
- `pool`은 여러 서버 묶음, `server`는 단일 서버 지정 — 지문이 단일 서버를 줬으니 `server` 사용
- 기존 pool 줄을 안 지우면 "지정된 서버로만 동기화"라는 요구사항을 완전히 만족 못 함
- `systemctl enable`만 하고 `--now` 빼먹으면 지금 당장은 안 켜짐

## Day 2 — 사용자 / 그룹 / 권한 / sudo

### 4. Setuid 개념 (4xxx)

#### 핵심 개념
실행 파일에 setuid(4000)를 걸면, 그 파일을 실행하는 사람이 누구든 **파일의 현재 소유자(owner) 권한**으로 프로세스가 동작함. 대표 예: `/usr/bin/passwd` (root 소유 + setuid → 일반 사용자도 `/etc/shadow` 수정 가능)

#### 적용
```bash
chmod u+s 파일        # 또는 chmod 4755 파일
ls -l 파일             # owner 실행 자리에 's' 표시 확인
find / -perm -4000 -type f 2>/dev/null   # 시스템 전체 setuid 파일 점검
```

#### 내가 헷갈렸던 것 / 배운 것
- **setuid는 "누가 실행할 수 있는가"를 결정하지 않음.** 실행 가능 여부는 별도의 일반 실행 권한(x 비트, group/other)이 결정하고, setuid는 "실행됐을 때 어떤 신분으로 도는가"만 결정
- `-rws------`(4700)처럼 group/other에 x가 없으면 소유자 본인 말고는 애초에 실행조차 못 함 → setuid 무의미
- **"파일 소유자"는 "만든 사람"이 아니라 현재 `ls -l`에 찍히는 소유자 필드(`chown`으로 언제든 바뀔 수 있는 동적 값)**. 예: `sudo mkdir`로 만든 디렉터리는 소유자가 root이지 실행한 사용자가 아님
- `s`(소문자) = setuid + x 있음(정상) / `S`(대문자) = setuid만 있고 x 없음(비정상, 무의미)



### 5. useradd vs usermod — sam 계정 쉘/만료일 변경

#### Question
```
Change the user sam's default login shell to /bin/bash.
Set the account to expire on December 21, 2029.
```

#### 잘못된 시도
```bash
useradd -s /bin/bash -e 2029-12-21 sam   # sam이 이미 존재하면 에러: user 'sam' already exists
```

#### 올바른 solution
```bash
usermod -s /bin/bash -e 2029-12-21 sam
```

#### Verification
```bash
getent passwd sam      # 쉘 필드가 /bin/bash 인지
chage -l sam            # Account expires 항목이 Dec 21, 2029 인지
```

#### 내가 헷갈렸던 것 / 배운 것
- 지문에 **"Change/Modify"**가 있으면 이미 존재하는 계정 → `usermod`. **"Create"**면 `useradd`
- `useradd`를 이미 있는 계정에 실행하면 에러만 뜨고 아무 반영도 안 됨 — 출력 메시지 확인하는 습관 중요
- 날짜 포맷은 `YYYY-MM-DD` 고정
- `-e`(계정 만료일)와 `-M`(비밀번호 최대 사용일수)는 다른 개념이라 혼동 주의

## Day 3 — SELinux / journald

### 6. SELinux Port 설정 (httpd 비표준 포트 82)

#### Question
```
A non-standard port 82 is used to run a web server that is having trouble providing service.
- Web server must serve all existing HTML files in /var/www/html (내용 변경/삭제 금지)
- Web server must be configured to serve on port 82
- Web server must start automatically at system startup
```

#### My Solution
```bash
# 0. httpd가 82를 듣도록 설정 (누락하기 쉬운 단계 — 반드시 명시)
vi /etc/httpd/conf/httpd.conf
#   Listen 80 → Listen 82 (또는 Listen 82 추가)

# 1. 진단
ls -Z /var/www/html/
semanage port -l | grep -w 82

# 2. 콘텐츠 컨텍스트 (필요할 때만 — /var/www는 기본 정책에 이미 httpd_sys_content_t 포함)
semanage fcontext -a -t httpd_sys_content_t "/var/www/html(/.*)?"
restorecon -Rv /var/www/html

# 3. SELinux 포트 정책
semanage port -a -t http_port_t -p tcp 82

# 4. 서비스 재시작 + 자동시작
systemctl restart httpd
systemctl enable httpd

# 5. 방화벽
firewall-cmd --permanent --add-port=82/tcp
firewall-cmd --reload

setenforce 1
getenforce
```

#### Verification
```bash
ss -tlnp | grep httpd
semanage port -l | grep -w 82
firewall-cmd --list-ports
curl http://localhost:82                      # 기본 환영 페이지가 아니라
curl http://localhost:82/실제파일명.html         # 실제 existing 파일명으로 직접 확인해야 진짜 검증됨
systemctl is-enabled httpd
```

#### 내가 헷갈렸던 것 / 배운 것
- **문제의 본질은 3개 독립된 층위**: httpd 자체 설정(Listen) / SELinux 포트 정책(semanage port) / 방화벽(firewalld). 하나만 빠져도 실패
- **`curl http://localhost:82`가 응답한다고 끝난 게 아님** — index.html이 없으면 RHEL/Rocky 기본 "Test Page"가 대신 뜨는데, 이건 "existing HTML files가 실제로 서빙된다"는 증명이 안 됨. 실제 파일명으로 직접 curl해야 진짜 검증
- `semanage port -a`는 그 포트가 **아무 타입에도 할당 안 되어 있을 때만** 동작. 이미 다른 타입에 할당돼 있으면 `-m`(modify) 사용. `-m`은 반대로 **로컬 규칙이 이미 등록되어 있을 때만** 동작 (`semanage fcontext -m /var/www/html/` 실행 시 `ValueError: File context ... is not defined` 에러로 직접 확인함)
- `restorecon -v` 실행했는데 출력이 없는 건 정상 — **변경사항이 있을 때만 출력**되는 것이지 에러가 아님. `/var/www`는 기본 정책에 이미 `httpd_sys_content_t`가 걸려있어서 애초에 바꿀 게 없었던 것
- `setenforce 0`으로 우회하면 진짜 설정이 맞는지 검증이 안 됨 — enforcing 상태에서 끝까지 해결해야 함
- **왜 SELinux 포트 정책이 필요한가**: httpd 프로세스는 `httpd_t` 도메인에서 동작하는데, 이 도메인은 `http_port_t`로 라벨된 포트에만 bind 가능. 82는 기본적으로 라벨이 없어서 방화벽을 다 열어도 httpd가 애초에 그 포트를 열지 못함 (방화벽=네트워크 레벨 통제, SELinux 포트 정책=프로세스 레벨 통제, 서로 다른 층위)


#### man 페이지
```bash
man semanage-port
man semanage-fcontext
man restorecon
```

### 6. SELinux Boolean 설정 (httpd_enable_homedirs)

#### Question
(공식 시험 범위 항목 — "Use boolean settings to modify system SELinux settings". 자체 실습)

#### My Solution
```bash
getsebool httpd_enable_homedirs
setsebool -P httpd_enable_homedirs on
```

#### Verification
```bash
getsebool httpd_enable_homedirs
semanage boolean -l | grep httpd_enable_homedirs   # default 값까지 확인 가능해서 더 정확
```

#### 내가 헷갈렸던 것 / 배운 것
- Boolean은 새 규칙을 만드는 게 아니라, **정책 안에 이미 컴파일되어 들어있는 조건부 허용 규칙의 스위치를 켜고 끄는 것.** 그래서 `semanage port/fcontext`처럼 규칙을 새로 등록하는 것보다 훨씬 가벼운 작업(재컴파일 불필요, 즉시 반영)
- 실제 커널이 참조하는 값은 `/sys/fs/selinux/booleans/httpd_enable_homedirs`
- `-P` 필수 — 빠뜨리면 재부팅 시 초기화됨 (시험 감점 포인트)

### 7. journald 영구 보존 설정

#### Question
(공식 EX200 Objectives — "Operate running systems" 도메인의 **"Preserve system journals"** 항목. 자체 실습)

#### My Solution
```bash
grep ^Storage /etc/systemd/journald.conf     # 변경 전 상태 먼저 확인 (sed 삽질 방지)
vim /etc/systemd/journald.conf
#   /Storage=auto  → 검색
#   :s/^#Storage=auto/Storage=persistent/  → 치환
#   :wq

systemctl restart systemd-journald
journalctl --flush
```

#### Verification
```bash
grep ^Storage /etc/systemd/journald.conf
# Storage=persistent

ls -la /var/log/journal/
# machine-id 기반 폴더 안에 .journal 파일 존재 확인

# 진짜 영구 보존인지는 재부팅으로 확인해야 확실함
reboot
journalctl --list-boots
# boot가 2개 이상(이전 부팅 로그 포함) 나오면 성공
```

#### 내가 헷갈렸던 것 / 배운 것
- 기본값 `Storage=auto`도 `/var/log/journal` 디렉터리만 있으면 persistent처럼 동작함. 진짜 차이는 "그 디렉터리가 없을 때 fallback 하느냐" — `persistent`는 없으면 강제로 만들어서 디스크 저장, `auto`는 없으면 메모리(`/run/log/journal`)에만 저장하고 재부팅 시 소실
- `systemctl restart`만으로는 기존에 메모리(`/run/log/journal`)에 있던 로그가 디스크로 안 옮겨질 수 있어서 `journalctl --flush`로 강제 반영. 단, 시험 채점 기준은 설정 파일+재시작까지만 볼 가능성이 높아 flush는 선택적으로 이해
- **검증의 본질은 재부팅 후 로그 생존 여부** — 설정 파일만 확인하고 끝내지 말고 `journalctl --list-boots`까지 확인해야 완전한 검증

## 원래 상태로 되돌리는 명령어 (Rollback)

```bash
# 1. YUM repo 원복
rm -f /etc/yum.repos.d/abc.repo
dnf clean all

# 2. 네트워크 원복 (enp0s3을 다시 DHCP/Automatic으로)

#  Activate a connection → enp0s3 재기동

# 3. chrony 원복
vi /etc/chrony.conf
#  server ntp.example.com iburst 삭제, pool 줄 주석 해제
systemctl restart chronyd

# 4. 사용자/그룹/권한 실습 정리
userdel -r sam 2>/dev/null
rm -rf /home/natasha/projects /shared

semanage port -d -t http_port_t -p tcp 82
firewall-cmd --permanent --remove-port=82/tcp
firewall-cmd --reload
systemctl disable --now httpd

# 6. SELinux Boolean 원복
setsebool -P httpd_enable_homedirs off

# 7. journald 원복
vi /etc/systemd/journald.conf
#  Storage=persistent → 다시 #Storage=auto로
systemctl restart systemd-journald
```
