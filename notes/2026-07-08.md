## 2-4. Resize LVM (Configure local storage)

### Question
Resize the logical volume VO and its file system to 230MiB. Make sure that partition is seldom exactly the same size requested, so a size within the range 217MiB to 243 MiB is acceptable

### My Solution
```bash
# 실패 재현 (lvextend는 확장 전용이라 축소 불가)
lvextend -r -L 230M /dev/wgroup/wshare
```
- `-r` : LV 크기 변경과 동시에 파일시스템 리사이즈까지 자동 처리
- `-L 230M` : 목표 크기를 **절대값** 230MiB로 지정 (`+230M`이면 현재 크기에서 상대적으로 추가하는 뜻이 됨)
- `/dev/wgroup/wshare` : 대상 LV 경로 (`/dev/VG명/LV명`)

```bash
# 정석: lvresize로 축소
lvresize -r -L 230M /dev/wgroup/wshare   # WARNING 뜨면 y
```
- 옵션 의미는 위와 동일
- `lvresize`는 목표가 현재보다 크면 확장, 작으면 축소를 **자동 판단** (`lvextend`는 확장만 가능)

### Verification
```
[lvextend 실행 결과]
Rounding size to boundary between physical extents: 232.00 MiB.
New size given (29 extents) not larger than existing size (50 extents)

[lvresize 실행 결과]
Reduced file system ext4 on wgroup/wshare.
Size of logical volume wgroup/wshare changed from 400.00 MiB (50 extents) to 232.00 MiB (29 extents).

[lvs 결과]
wshare wgroup -wi-ao---- 232.00m
```
232MiB → 217~243MiB 범위 안 → 정답 인정 범위.

### 내가 헷갈렸던 것 / 배운 것
- `lvextend`는 확장 전용, 목표가 현재보다 작으면 에러남
- `lvresize`는 축소 시 내부적으로 unmount → e2fsck → resize2fs → remount 순서로 안전하게 처리
- PE 8MiB 단위라 230÷8=28.75 → 정확히 안 나눠떨어져서 29 extent(232MiB)로 반올림됨
- ext4는 축소 가능하지만 xfs는 축소 불가 (재확인 필요)

## 2-5. LVM 생성 (PE 8MB, 50 extents, ext4, fstab)

### Question
Create an LVM name wshare from wgroup volume group. Note the following: PE size should be 8MB, LVM size should be 50 extents. Format with "ext4" file system and mount it under /mnt/wshare. And it should auto mount after next reboot.

### My Solution
```bash
fdisk /dev/sdb
```
- `fdisk /dev/sdb` : 대상 디스크(`sdb`)를 지정해서 파티션 편집 도구 진입 (변경사항은 `w` 치기 전까지 메모리상에만 존재)

```
n        # 새 파티션 생성
p        # primary 선택 (MBR 방식은 주파티션 최대 4개 제한 때문에 구분)
1        # 파티션 번호
Enter    # 시작 섹터 기본값 (fdisk가 안전하게 자동 정렬)
+500M    # 끝 위치 지정 — 요구 용량(400M)보다 여유있게 500M (메타데이터 오버헤드 감안)
t        # 파티션 타입 변경 모드 진입
L        # 타입 리스트 출력 (MBR/GPT마다 코드가 달라서 외우지 말고 항상 확인)
lvm      # "Linux LVM" 타입 alias 입력 (hex코드로는 8e와 동일)
w        # 변경사항을 디스크에 실제로 저장
```

```bash
pvcreate /dev/sdb1
```
- 파티션을 LVM이 인식할 수 있는 **물리 볼륨(PV)**으로 초기화 (LVM 전용 메타데이터 기록)

```bash
vgcreate -s 8M wgroup /dev/sdb1
```
- `-s 8M` : PE(Physical Extent, 최소 할당 단위) 크기를 8MiB로 지정 — **VG 생성 시점에만** 결정 가능, 이후 변경 불가
- `wgroup` : 새로 만들 VG 이름 (임의 지정)
- `/dev/sdb1` : 이 VG에 포함시킬 PV

```bash
lvcreate -l 50 -n wshare wgroup
```
- `-l 50` : extent **개수**를 50개로 지정 (8MiB×50=400MiB). `-L`은 절대 용량 단위, `-l`은 개수/비율(`%FREE` 등) 단위
- `-n wshare` : 새로 만들 LV 이름
- `wgroup` : 어느 VG에서 잘라올지 지정

```bash
mkfs.ext4 /dev/wgroup/wshare
```
- LV를 ext4 파일시스템으로 포맷

```bash
mkdir /mnt/wshare
```
- 마운트할 디렉토리(마운트 포인트) 생성

```bash
echo "/dev/wgroup/wshare /mnt/wshare ext4 defaults 0 0" >> /etc/fstab
```
- `>>` : 기존 fstab 내용은 보존하고 맨 아래에 한 줄 **추가**
- 필드 순서: `장치경로  마운트포인트  파일시스템타입  마운트옵션  dump여부  fsck순서`

```bash
mount -a
```
- fstab에 있는 항목 중 **아직 마운트 안 된 것**을 전부 마운트 (재부팅 전 fstab 문법 오류를 미리 검증하는 용도)

### Verification
```
[vgdisplay wgroup] PE Size    8.00 MiB
[lvdisplay wgroup/wshare] Current LE   50
[df -h /mnt/wshare]
/dev/mapper/wgroup-wshare  365M   14K  341M   1% /mnt/wshare
```

### 내가 헷갈렸던 것 / 배운 것
- 파티션 타입 코드(`8e`)는 MBR/GPT에 따라 다르게 매겨질 수 있음 → `t → L`로 항상 확인 후 선택
- 파티션을 요구 크기(400M)보다 여유(500M)있게 잡은 이유: LVM 메타데이터 오버헤드 + 섹터 정렬 때문에 딱 맞추면 extent가 모자랄 수 있음
- `vgcreate -s`(PE 크기)는 VG 생성 시점에만 결정 가능
- `lvcreate -l`(개수) vs `-L`(절대 용량) 구분
- `/dev/sdb`와 `/dev/sdb1`이 같은 하드웨어 ID 공유해서 뜨는 WARNING은 무시 가능

## 2-6. Swap 파티션 (fdisk → mkswap → swapon → fstab)

### Question
(원문 미확보 — Geuni/dump 요약 기준) Swap용 파티션을 생성하고 mkswap/swapon으로 활성화한 뒤, 재부팅 후에도 유지되도록 fstab에 등록하라.

### My Solution
```bash
fdisk /dev/sdb
```
```
n        # 새 파티션 생성
p        # primary 선택
2        # 파티션 번호 (1번은 이미 wshare용으로 사용 중)
Enter    # 시작 섹터 기본값
Enter    # 끝 섹터 기본값 = 남은 공간 전부 사용
t        # 타입 변경 모드
2        # 대상 파티션 번호 지정 (디스크에 파티션이 2개라 어느 걸 바꿀지 선택)
swap     # "Linux swap" 타입 alias (hex 82와 동일)
w        # 저장
```

```bash
mkswap /dev/sdb2
```
- 해당 파티션에 **swap 전용 포맷**(시그니처) 적용. 일반 파일시스템 포맷이 아니라 swap 용도로만 쓰겠다는 표시

```bash
swapon /dev/sdb2
```
- `mkswap`은 포맷만 한 상태라 아직 안 쓰이는 중 — `swapon`이 커널에 "이 영역을 실제로 swap으로 써도 좋다"고 활성화 지시

```bash
echo "/dev/sdb2   swap   swap   defaults   0 0" >> /etc/fstab
```
- 필드 순서: `장치  마운트포인트(swap은 해당없음, 그냥 swap 표기)  타입(swap)  옵션  dump  fsck`

### Verification
```
[free -h]
Swap:          6.5Gi          0B       6.5Gi

[swapon --show]
/dev/dm-1 partition   2G   0B   -2
/dev/sdb2 partition 4.5G   0B   -3
```

### 내가 헷갈렸던 것 / 배운 것
- `mkswap`(포맷)과 `swapon`(활성화)은 별개 단계
- fstab 등록 안 하면 재부팅 시 swap 사라짐

## 2-7. tuned (Configure recommended tuned profile)

### Question
Configure recommended tuned profile.

### My Solution
```bash
systemctl enable --now tuned
```
- `enable` : 부팅 시 자동 시작되도록 등록
- `--now` : 등록과 동시에 **지금 바로** 서비스 시작까지 한 번에 처리

```bash
tuned-adm recommend
```
- 현재 하드웨어/가상화 환경을 자동 감지해서 **추천 프로파일**을 조회 (지금 활성화된 것과는 별개 개념)

```bash
tuned-adm profile virtual-guest
```
- 추천받은 프로파일을 **명시적으로 적용** (자동 감지로 이미 같은 값이어도, 이 명령으로 `/etc/tuned/active_profile`에 명시적으로 기록해두는 것이 중요)

```bash
reboot
```
- 설정이 재부팅 후에도 유지되는지 지속성 확인용

```bash
tuned-adm active
```
- 현재 적용된 프로파일 확인 (재부팅 후에도 `virtual-guest` 유지되는지 최종 검증)

### Verification
```
[재부팅 후]
tuned-adm active
Current active profile: virtual-guest
```

### 내가 헷갈렸던 것 / 배운 것
- `tuned-adm recommend`(추천값) vs `tuned-adm active`(현재 적용값) 개념 차이
- 자동 감지된 값이 이미 추천값과 같아도, 명시적으로 설정해둬야 함
- 재부팅 후 지속성 확인이 검증의 핵심

## 원래 상태로 되돌리는 명령어 (Rollback)

```bash
# --- LVM/파티션 되돌리기 ---
umount /mnt/wshare
lvremove wgroup/wshare
vgremove wgroup
pvremove /dev/sdb1
fdisk /dev/sdb    # d -> 1 -> w (파티션 삭제)

# --- Swap 되돌리기 ---
swapoff /dev/sdb2
fdisk /dev/sdb    # d -> 2 -> w
# /etc/fstab 에서 wshare, sdb2 관련 줄 직접 삭제 (vi /etc/fstab)

# tuned는 기본값(virtual-guest)과 동일하므로 되돌릴 필요 없음
```
